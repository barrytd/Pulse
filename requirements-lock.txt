# Captured via `pip freeze` on 2026-05-14 against Python 3.14 on Windows
# x64, with three CVE-fix bumps applied:
#   - pip:              25.3   -> 26.1.1   (CVE-2026-1703, -3219, -6357)
#   - pytest:           9.0.2  -> 9.0.3    (CVE-2025-71176)
#   - python-multipart: 0.0.26 -> 0.0.28   (CVE-2026-42561)
#
# Use this for production / hosted deploys where reproducible builds
# matter: a fresh `pip install -r requirements-lock.txt` six months from
# now gets the same byte-for-byte versions we ran on the CI green tests.
#
# Why both files exist:
#   requirements.txt        — `>=` ranges. Lets `pip install -r` pull
#                              fresh minor patches during dev; survives
#                              Python version bumps and upstream
#                              re-releases without manual maintenance.
#   requirements-lock.txt   — exact `==` pins. Production deployments
#                              (Render service, customer-self-host) use
#                              this so a compromised or buggy upstream
#                              update can't silently break Pulse or
#                              introduce a vulnerable transitive dep.
#
# To refresh after a verified upgrade:
#   1. pip install -r requirements.txt          # pull latest within ranges
#   2. python -m pytest -q                       # confirm everything green
#   3. python -m pip_audit --strict              # confirm no new CVEs
#   4. pip freeze > requirements-lock.txt        # regenerate this file
#   5. commit + push (CI re-runs against the fresh pins)
#
# Includes every package currently installed in the dev environment
# (runtime + test + build deps). Production deploys end up with pytest +
# pyinstaller too — slightly larger image, zero security cost.
